01

为什么是这一类项目

普通项目代码在本机跑，写对了基本就对了。这一类的三个结构性特征，让“代码看着对”和“线上真的对”之间裂开一条鸿沟——AI（和人）最贵的三个错误由此而来。

TRAP_01

真相不在代码里

配置中心、字典码值、组织维度、有没有数据库迁移框架——AI 读穿代码也推不出来，只能被告知或去现场探。

表现 › 合法 API 也可能取错值；目标环境迁移缺失时加列即崩。
TRAP_02

改完 ≠ 生效

代码不只在本机生效，旧码 / 旧包 / 旧缓存可藏在任意一环。

表现 › 本地未同步远端→CI 编旧码；部署 SUCCESS 但目标环境仍是旧版本。
TRAP_03

行为由配置决定

流程引擎 / 配置中心 / 权限系统——很多“bug”根本不是代码问题，写代码去修反而引入新 bug。

表现 › 按钮占位符＝配置未发布；外部流程不回调＝服务或规则未生效。
02

脊柱 · 四个阶段

一次任务的生命周期，按 0 → 1 → 2 → 3 顺序按阶段执行，不跳步。每一阶段堵一类最贵的错误。

0
归因
Attribute
动码前先钉到 代码 / 配置 / 运维 / 数据 某一层并给证据。不是“代码”层就不许写代码。
治 › 为非代码问题写代码（最贵的浪费）
1
外部真相
Ground Truth
结构性真相入档案（随仓库走），运行时状态现场探针。绝不把易变状态写死进文档。
治 › 不知真相就动手、取错值
2
取值地图
Resolution Map
每个外部值钉到唯一正路。不写死 ID ≠ 只是别写裸数字——走真 API 也会取错值。
治 › 走对路的 API 仍串值 / 静默变空
3
验证闭环
Verify · 第一枪
按改动类型取线上实证。无实证，不许说“修好了”。Evidence before assertions.
治 › 假成功上线
↺ 回灌：阶段 0/3 抓到新真相，当场写回档案——AI 抄写、人审、合并。档案是活体，不是死文档。
03

阶段 2 · 不写死 ID 的真正含义

挑战一个直觉误区：很多人以为“不写死 ID”＝别写裸数字、改成查字典。这个理解不够，甚至有害。真正的杀手不是“写了字面量”，而是“取值走了一条看似合理、实则错误的路”。

歧路 · 走了真 API 却取错值
// ❌ 扁平查多级字典树 → 数字 value 跨字典复用
String name = codeUtils.getNameByParentType(
    parentType, value);
// 多级树 + value 复用 → 实测显示成另一个类目

// ❌ 查通用字典取状态名
String s = codeUtils.getCodeName("biz_status", code);
// 码值对不上 → 静默变空
正路 · 钉到唯一正路
// ✅ 按层级查（专用 Resolver）
String name = typeNameResolver
    .resolve(parent, child);
// 走层级，不串值

// ✅ 用枚举，不查字典
String s = BizStatusEnum.getName(code);
// 码值自洽，不会静默变空

机制 › grep 魔法数字的 lint 根本抓不到这两个坑——它们走的是真·API。所以阶段 2 的中心是一份 「取值地图」：每个外部概念钉到唯一正路，表里没有的停下问人，绝不自创查法；review 查的是“有没有绕开正路”，而非“有没有裸数字”。

04

阶段 3 · 改动类型 → 必需实证

该验证哪几环，取决于你改了什么。这是密度最高的痛点，也是落地第一枪。

改动类型
必需实证
不做的后果
后端
git ls-remote origin == 本地 HEAD + 版本/业务探针验目标环境
未同步远端→CI 拉旧码，部署 SUCCESS 全是旧逻辑
前端
清缓存重登后线上调接口验证
浏览器缓存旧 bundle，“看不到改动”（已坑两次）
DB 列
information_schema 实查列存在 · 先手工 ALTER
无迁移框架，引用不存在的列部署即崩
流程引擎
手动 POST 回调 → 状态翻转 + 编辑器 i18n / 变量已配
以为链路通、其实引擎没发；按钮显示占位符
铁律 / IRON LAW

“修好了 / 部署成功 / 应该没问题 / 已完成”这类话，没有引用对应实证，一律不许说出口。拿不准时默认“未验证”，把缺的实证列出来——绝不用“应该 / 理论上”补位。

05

活体机制 · 防腐烂

档案一旦冻结在第一天就开始过期。必须有机制把“刚踩的坑”自动变成“档案里的永久条目”。

踩坑即回灌

  1. 谁发现谁回灌 › 阶段 3 抓到假成功、或阶段 0 查出非代码真因，当场写回对应资产。
  2. AI 当抄写员 › 任务收尾，AI 主动产出“本次新学到的外部真相”草稿——它刚踩完坑，最便宜的记录者。
  3. 人审是阶段 › 出口从“个人机 memory”改成“项目 PR”，人审一下再合并，防写错 / 写冗余。
  4. review 卡点 › reviewer 必问“这次学到新外部真相了吗？入档了吗？”——没回灌的 PR 不算完。
