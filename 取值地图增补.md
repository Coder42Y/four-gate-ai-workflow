# 取值地图增补 · 给 code-self-review 加 §6.9

> **为什么单独加一节**：`code-self-review` 现有 §6.7「硬编码红线」通常只抓**裸字面量**。但很多高危取值坑是**走了真 API 却取错路**：例如用扁平查询去查层级字典、用不匹配的字典查业务状态、用错误组织维度取审批人。这些 grep 裸数字**完全抓不到**（它们是合法的方法调用）。
>
> **机制**：把「取值地图」的**禁用歧路**变成一张 banned-resolver 命中表——grep **禁用方法名**，命中即违规。检测的是"有没有绕开正路"，不是"有没有裸数字"。
>
> **怎么用**：推荐执行 `bash deploy.sh --install-review-addon` 自动把本增补写入 `code-self-review/SKILL.md` 的受管块。也可以手工把下面 §6.9 整段并入 `code-self-review` 的 §六之后，并在 §七「常见违规高频清单」追加对应一键扫命令。取值地图本体（正路/歧路全表）以 `ai-workflow/环境真相档案.md` §四为 SSOT。

自动安装会查找：
- `~/.agents/skills/code-self-review/SKILL.md`
- `~/.claude/skills/code-self-review/SKILL.md`
- `~/.codex/skills/code-self-review/SKILL.md`（旧版兼容路径）

如果你的 skill 在其他位置，可指定：

```bash
CODE_SELF_REVIEW_SKILL=/path/to/code-self-review/SKILL.md bash deploy.sh --install-review-addon
```

## 6.9 取值绕路红线（取值地图 SSOT：环境真相档案 §四）⚠️ 高危专题

> 凡"取外部值走了禁用歧路"全是 ❌ **违规**，必改。命中后给出正路。本节专抓 §6.7 抓不到的"真 API 取错值"。

| 检测 | 命令 / 模式 | 命中判定 | 正路 |
|------|-------------|---------|------|
| 字典名扁平查（跨级串值） | `git grep -nE 'getCodeNameByParentCodeType' -- 'backend-module/*'` | ❌ | `selectByParentCodeType` 按层级（`TypeNameResolver`） |
| 业务状态名查错字典（静默变空） | `git grep -nE 'codeUtils.*(status\|getCodeName)' -- 'backend-module/*'` 后人工核对是否在取业务状态名 | ⚠️ 错字典则 ❌ | 项目取值地图中的状态枚举/状态转换器 |
| 审批人单维度取岗（找不到人） | `git grep -nE '(APPROVER_ROLE\|selectApproverIds)' -- 'backend-module/*'` 后核对是否带回退 | ⚠️ 无回退则 ❌ | `selectApproversWithFallback`（部门→签约主体回退） |
| 全量组织却用受限子集 | `git grep -nE 'UserScopedDepartmentApi' -- 'frontend-app/src/*'` 后核对该处是否需"全量组织架构" | ⚠️ 需全量却用它则 ❌ | 页面侧不带 userId 调 `getOrganTree` 取 `orgType=3` |
| 选择器带出字段未往返 | `git grep -nE '(setOrgan\|setSignEntity\|选人回调).*(Name\|Id)' -- 'frontend-app/src/*'` 后核对 payload/VO/loadDetail 是否齐 | ⚠️（人工确认完整往返） | payload+列+VO+loadDetail 全补 |
| 新概念自创查法 | `[human-only]`：出现"取某外部名/某外部 ID"且未走环境真相档案 §四任一正路 | ⚠️ 标"取值地图未覆盖，先问人，勿自创" | 停下问人，补进取值地图 |

**反例（必改）：**

```java
// ❌ 走了真 API，但扁平查多级字典树 → 数字 value 跨字典复用 → 串值
String name = codeUtils.getCodeNameByParentCodeType(parentType, value); // 多级字典被扁平查，可能串值
// ✅ 按层级
String name = contractTypeNameResolver.resolve(big, small); // selectByParentCodeType

// ❌ 查 foundation 字典取状态名 → 码值对不上 → 静默变空
String status = codeUtils.getCodeName("some_status_dict", code);
// ✅ 用枚举
String status = BusinessStatusEnum.getName(code);

// ❌ 单维度取岗 → 维度不兼容 → 找不到审核人
List<Long> ids = roleUserMapper.selectApproverIds(deptId, "APPROVER_ROLE");
// ✅ 带回退
List<Long> ids = approverService.selectApproversWithFallback(contractDeptId, signEntityId, "APPROVER_ROLE");
```

**判定原则**：
- 这些是"看着合规、运行才错"的坑——**拿不准时不要放过，标 ⚠️ 并写"待人工确认是否走正路"**，比漏报安全。
- 命中后**必须在报告里写出正路**（指向环境真相档案 §四对应行），不能只说"违规"。

## §七 一键扫追加命令（cmd 友好）

在 `code-self-review` §七末尾追加：

```bash
# === 取值绕路红线（§6.9）===
git grep -nE 'getCodeNameByParentCodeType' -- 'backend-module/*'
git grep -nE 'codeUtils.*(status|getCodeName)' -- 'backend-module/*'
git grep -nE 'UserScopedDepartmentApi' -- 'frontend-app/src/*'        # 命中后核对是否需全量组织
git grep -nE 'APPROVER_ROLE' -- 'backend-module/*'           # 命中后核对是否带 fallback
```

每条预期**零命中或命中后人工核对走的是正路**；扁平查字典/查 foundation 状态名**任何命中即违规候选**。

## 维护约定

- **取值地图本体只有一份**：`环境真相档案.md` §四。本增补只是"把歧路变成可 grep 的检测"。新增正路/歧路先改档案，再回来加检测行。
- 每条新检测都要能指回档案对应行（SSOT 单点），避免规则散落漂移——这正是"取值地图"自己的精神：别重复造、钉到唯一正路。
