## 一、部署链路真相（最致命，先读这条）

> 母题：**"改完了" ≠ "生效了"**。代码不在本机跑，旧东西可藏在任意一环。

| 环节 | 脱敏真相 | 铁律 |
|---|---|---|
| 代码同步 | Jenkins 从 **内部 Git 远端**拉代码编译，**不是本地** | 触发构建前先 `git ls-remote origin <branch>` 比对本地 `git rev-parse HEAD`；不一致=没 push，构建必是旧码 |
| 构建 | `backend-service-build` 走 Jenkins job `backend-service-build`（crumb+cookie 同会话 POST `/job/backend-service-build/build`） | 轮询构建号 + `result=SUCCESS` + 读 `consoleText`，别只看"触发成功" |
| 部署 | 部署 SUCCESS **≠** jar 真换新 | 用 typeName 探针验 jar 新旧（见 §五） |
| 重启 | 目标主机 部署机制 | **别用 `pkill -f backend-service.jar`**（会自杀 exec shell） |
| 前端缓存 | index.html **无缓存头**，浏览器缓存旧 bundle | 复验前必清缓存 + 重登（已坑两次：工单 A/B） |
| token | 详情接口 token **~5min 过期** → 页面整片变空(401) | 页面突然空 ≠ 改坏了，先重新登录再判 |

来源：`example-case` / `example-case` / `example-case`

## 二、数据库真相

| 真相 | 说明 | 铁律 |
|---|---|---|
| **目标环境无自动迁移** | 历史表是手工/dump 进库，迁移脚本不会自动跑 | "加列 + 代码引用该列"的 commit 部署即崩 BUSINESS_ERROR_CODE（mapper 查不存在的列）。**上代码前先在目标环境手工执行迁移补列** |
| **目标环境 MySQL 方言需实测** | 不支持 `ADD COLUMN IF NOT EXISTS`（那是 MariaDB 专有） | 用普通 `ADD COLUMN` 一次性跑，否则报 1064 |
| 不要拿仓库新 V 脚本跑目标环境的基础数据 | 会把已干净的码值改脏（工单 A 码值对不上/工单 B 小类空/settlement_type 空，是实查发现） | 目标环境无自动迁移历史表，无 checksum 风险也不需 repair，但**别乱跑迁移** |
| 诊断"系统异常"先查列在不在 | `information_schema.columns` 实查 | 加列类改动的第一排查项 |

来源：`example-case` / `example-case` / `example-case`

## 四、取值地图（闸门 2 本体）⚠️ 核心

> **不写死 ID 的真正机制不是"禁裸数字"，是"钉到唯一正路"。** 下面每个外部概念只有一条正路；走歧路即使是真·API 调用也会取错值（串值/静默变空）。**表里没有的概念，停下来问人，绝不自己发明查法。**

| 外部概念 | ✅ 唯一正路 | ❌ 禁用歧路 | 走歧路的后果 |
|---|---|---|---|
| **appId** | 常量 `main-app=<从配置读取>` / 配置 | 裸数字、抄别的应用 | 打到别的应用 |
| **合同类型名（大类/小类）** | `selectByParentCodeType` 按层级查（`TypeNameResolver`，3 个 Wrapper 复用） | `codeUtils.getCodeNameByParentCodeType` 扁平查 | 多级树 + 数字 value 跨字典复用 → 串值，type=A 类别 A 实测变"类别 B"（Jira 工单 D） |
| **合同状态名** | `BusinessStatusEnum.getName` | `codeUtils` 查 foundation `contract_status` 字典 | 码值对不上，**静默变空**（Jira 工单 C / 合同变更两处已踩） |
| **审批人（按岗位）** | `selectApproversWithFallback`（合同部门→签约主体回退） | 单维度按岗取人 | 岗位组织维度不兼容→"找不到审核人"流程卡死（Jira 工单 E/F） |
| **部门（全量组织架构）** | 页面侧不带 userId 调 `getOrganTree` 取 `orgType=3` 填 el-select | `UserScopedDepartmentApi` 按登录人 userId（那是受限子集） | 只能看到自己的子集，要全量却给了片段 |
| **签约主体 / 部门 取数** | `orgType=2`=签约主体、`orgType=3`=部门 | 混用 | 取错维度 |
| **菜单/按钮权限** | UIM funcPerm（appId=<从配置读取>）+ 前端既有 `$hasBtnPermission`（读 route.meta.buttons），菜单走动态路由 | 新建 v-hasPerm、写死菜单 ID | 权限不生效/写死 |

来源：`example-case` / `example-case` / `example-case` / `example-case` / `example-case` / `example-case`

> **通用坑**：前端"选择器带出的展示字段"必须查**完整往返**（payload/列/VO/loadDetail 全要有），否则保存后详情/编辑丢失（责任人变更所属公司/部门曾整条丢，根因=值只在前端临时态没落库，`example-case`）。

## 五、流程引擎（ezgo/工作流）真相

> 母题：**流程里大半"bug"是配置/运维，不是代码**（见闸门 0）。

| 真相 | 说明 |
|---|---|
| runProcess 调用必须带 token | 裸 axios 无 token → 点同意/拒绝必 401，引擎不前进。修法=全局 axios 拦截器（这条**是代码**） |
| 审批按钮中文标签靠编辑器配 | 占位 key 的正路=流程编辑器**逐节点勾命令按钮再保存发布**，引擎才生成中文 i18N（**配置层**，不是代码） |
| 网关流程变量需注入 | 否则 `No such property isSubCompany` 流程卡死（**配置层**） |
| 引擎不自动发 process-end 回调 | 回写链路代码/网关白名单/handler 都对；卡点是引擎不自动发（事件派发服务停摆，**运维层**）。手动 POST 回调→APPROVED 可证链路通 |
| 合同申请发号 | 靠 XXL 定时任务兜底；根因=流程平台 `contract_apply_process`「事件」配置为空 + 业务系统缺少对应回调端点。已验证平台支持 http 事件回调 |
| 岗位 code 表 / isSubCompany 判定 | 取 APPROVER_ROLE 岗要兼容公司级/部门级维度（见取值地图） |

**typeName 探针（验后端 jar 新旧的标准动作）**：开"类别 A 合同(typeA)"详情，`typeName`="类别 A"=新 jar，="类别 B"=旧 jar。
