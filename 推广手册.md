# 四阶段 AI 工作流 · 推广手册

> 给团队负责人和开发同事：为什么要接入、怎么三天起步、怎么证明有效、怎么回应常见质疑。

---

## 1. 30 秒说明

AI 写代码本身不难，难在真实业务项目里它经常缺三类上下文：外部配置和数据真相、部署链路真相、完成标准真相。

这套工作流把这些约束做成 Codex skills + 项目 `AGENTS.md` 路由。开发者仍然用自然语言提需求，Codex 在合适时机自动进入方案发散、澄清、归因、取值地图、验证闭环和回灌，不要求大家背阶段号。

---

## 2. 它解决的具体问题

| 现场问题 | 常见真因 | 工作流怎么挡 |
|---|---|---|
| 改了代码，目标环境还是旧表现 | 本地没 push、CI 编旧码、部署包没换、前端缓存旧产物 | 收尾必须走 `verify-closure`，远端 HEAD、构建号、版本/业务探针缺一项就不说完成 |
| 页面值、状态名、审批人看起来随机错 | 调了真实 API，但外部值取值路径错 | 建取值地图，把每个外部概念钉到唯一正路，review 查禁用歧路 |
| 按钮、流程、权限、回调异常 | 根因在配置、权限、流程节点、外部服务，不在代码 | 动码前 `attribute-rootcause` 先定层；非代码层输出动作清单，不写代码绕过去 |
| 只有一个粗想法，直接开工容易做偏 | 目标、用户、方案边界和取舍标准还没定 | `ai-task-brainstorm` 先出候选方案和推荐路径，不写代码 |
| 需求一句话扔给 AI，做完发现方向不对 | 目标、边界、验收、风险没收敛 | `ai-task-preflight` 先形成任务简报，能查的信息由 AI 自己查 |

## 3. 日常触发方式

| 开发者自然说法 | Codex 应做的事 |
|---|---|
| “帮我想想这个功能怎么设计，先别写代码” | 发散 2-4 个方案并给推荐路径 |
| “帮我实现这个需求，但你先看看还缺什么信息” | 收敛任务简报 |
| “线上没生效，先别急着改代码” | 归因到代码/配置/运维/数据 |
| “这里要取业务状态名，别走错字典” | 查取值地图 |
| “验证一下这次改动能不能交付” | 出验证闭环报告 |
| “这个坑下次别再踩，帮我记到项目里” | 起草回灌候选 |

阶段名仍保留在蓝图里，用于解释机制和做审计；日常操作靠这张路由表。

## 4. 三天起步路线

**Day 1：接入 Codex 工作流**

在每个业务 repo 根目录执行：

```bash
curl -fsSL https://raw.githubusercontent.com/Coder42Y/four-gate-ai-workflow/master/bootstrap.sh -o /tmp/four-stage-bootstrap.sh
bash /tmp/four-stage-bootstrap.sh
```

然后从业务 repo 根目录启动 `codex`。安装会完成三件事：

- 全局安装 `ai-task-brainstorm`、`ai-task-preflight`、`attribute-rootcause`、`verify-closure`。
- 生成项目 `ai-workflow/`。
- 注入 `AGENTS.md` / `CLAUDE.md` 的自然语言路由。

**Day 2：补项目事实**

编辑 `ai-workflow/环境真相档案.md` 的 `[待填]` 项，优先补：

- 部署链路：远端分支、CI、构建命令、部署方式、日志入口、版本探针。
- 数据库：迁移机制、加列流程、只读确认 SQL。
- 外部系统：appId/tenantId/权限系统/配置中心/流程引擎入口。

**Day 3：建立取值地图**

把高频外部值写进 `环境真相档案.md` §四：

- 业务状态、业务类型、角色/岗位、组织/主体、菜单/按钮权限。
- 每个概念写唯一正路、禁用歧路、走歧路后果。
- 有 `code-self-review` 时执行 `bash deploy.sh --install-review-addon`，把禁用歧路接进 review。

## 5. 怎么量化效果

| 指标 | 统计方式 | 可汇报口径 |
|---|---|---|
| 假成功拦截次数 | `verify-closure` 报告里“未验证完成”的次数 | 本周拦下 N 次旧码/旧包/旧缓存风险 |
| 非代码归因次数 | 归因结论落到配置/运维/数据的次数 | N 次避免了为配置或环境问题写代码 |
| 取值歧路命中数 | `code-self-review` §6.9 命中数 | N 个串值/变空类风险在 review 阶段被发现 |
| 档案回灌数 | 回灌候选合并数 | 项目外部真相持续沉淀，减少重复踩坑 |
| 方案收敛记录数 | brainstorming 结论里有推荐路径的次数 | 粗想法先比较再开工，减少方向性返工 |
| 任务简报覆盖率 | 模糊需求进入开发前是否有简报 | 减少返工和验收口径漂移 |

## 6. 回应常见质疑

| 质疑 | 回应 |
|---|---|
| “是不是又多一套流程？” | 日常不用背流程。Codex 读 `AGENTS.md` 后按自然语言触发，开发者照常描述任务。 |
| “我们项目技术栈不一样” | skills 解决的是通用协作动作；环境真相档案和取值地图按项目自填。 |
| “会不会拖慢开发？” | 它拦的是高成本返工：旧码验证、非代码问题写代码、取值走错路。小任务足够清楚时不会硬追问。 |
| “AI 写档案会不会写错？” | AI 只写回灌候选，人审后合并。档案是项目资产，不是个人记忆。 |
| “为什么还叫四阶段？” | 四阶段是内部审计模型；对开发者暴露的是自然语言路由和四个 skills。 |

## 7. 落地清单

- [ ] 每个业务 repo 根目录执行 bootstrap。
- [ ] 从业务 repo 根目录启动 `codex`，确认 `AGENTS.md` 被读取。
- [ ] 填 `ai-workflow/环境真相档案.md` 的部署链路、DB、外部系统入口。
- [ ] 建立取值地图，并把高危歧路接入 `code-self-review`。
- [ ] 团队约定：没有可复核实证，不说“修好了”。
- [ ] Review 时问一句：这次有没有新外部真相需要回灌？

---

一句话总结：把 AI 从“直接写代码的助手”，变成“先发散收敛方案、先形成任务简报、先归因、先核外部真相、按正路取值、拿证据收口、把坑沉淀回项目”的开发协作者。
