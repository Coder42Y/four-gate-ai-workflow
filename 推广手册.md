# 四阶段工作流 · 推广手册（给人看）

> 给团队负责人和同事：**为什么要这套、怎么三天起步、怎么证明它有用、怎么回应质疑。**
> 技术规范在 `四阶段蓝图.md` 及各 Skill，本手册只讲"推广"。

---

## 1. 电梯陈述（30 秒）

> AI 写代码本身不难，难在真实业务项目里反复翻车的四件事：**需求没问清、外部真相不知道、为不是代码问题的问题写代码、没实证就说"修好了"。** 四阶段加一个开工前准备 skill，就是把这些高成本错误做成 AI 每次自动遵守的纪律。不是又一份没人看的规范——是让"做对"成为最省事的路径。

---

## 2. 三个真实代价（讲给怀疑的人听）

这类问题在很多业务系统里都会出现：

- **"改了三遍目标环境还是旧的"** —— 本地 commit 没同步到远端，CI 构建的仍是旧代码；部署显示 SUCCESS，但实际服务没更新。**白白来回三趟。**
- **"页面显示值和数据库真实值对不上"** —— 代码走了正规查询，也没有写死 ID，却因为跨层取值路径不一致把值串了。**没有取值地图，"不写死"也救不了。**
- **"按钮或流程状态异常，写了一下午代码没用"** —— 根本不是代码问题，而是配置、权限、流程节点或外部服务没配好。**为非代码问题写代码 = 纯浪费。**

每一条都是返工成本和交付风险。四阶段各堵一类。

---

## 3. 四阶段一句话版

| 前置 | 一句话 | 治哪个代价 |
|---|---|---|
| **准备 任务简报** | 需求模糊时先追问：目标、边界、证据、验收是什么？ | 没准备好就让 AI 动手 |

| 阶段 | 一句话 | 治哪个代价 |
|---|---|---|
| **0 归因** | 动码前先问：代码/配置/运维/数据，是哪层？不是代码就别写代码 | 为非代码问题写代码 |
| **1 外部真相** | 代码里看不到的真相（迁移机制、appId、字典层级）先核对 | 不知真相就动手 |
| **2 取值地图** | 每个外部值钉到唯一正路；不写死 ≠ 只是别写裸数字 | 走真 API 取错值 |
| **3 验证闭环** | 没线上实证，不许说"修好了" | 假成功上线 |

## 4. 三天起步路线（最小可用）

**别一上来铺全套。按这个顺序拿第一个胜仗：**

**Day 1 — 一键部署 + 立铁律**
- 第一次使用先安装 Claude Code 命令入口：
  `curl -fsSL https://raw.githubusercontent.com/Coder42Y/four-gate-ai-workflow/master/install-claude-command.sh -o /tmp/four-stage-install-claude-command.sh`
  `bash /tmp/four-stage-install-claude-command.sh`
- 之后每个新业务 repo：进入项目根，启动 `claude`，输入 `/four-stage-install`。
- 全局 Skills（Claude + Codex）+ 项目级模板（环境真相档案、取值地图指南、CLAUDE.md 注入）一步到位。
- 团队约定：模糊任务先用 `ai-task-preflight` 形成任务简报。
- 团队约定一句话铁律：**"没有线上实证，不说修好了"**。
- 部署后编辑 `ai-workflow/环境真相档案.md` 中的 `[待填]` 项。

**Day 2 — 完善底座（环境真相档案）**
- 填写 `ai-workflow/环境真相档案.md` 的剩余 `[待填]` 项。
- 重点：部署链路、数据库真相、appId 权威表。
- 让 AI 每次会话自动读取。

**Day 3 — 建立取值地图**
- 按照 `ai-workflow/取值地图指南.md` 的引导，梳理本项目的取值地图。
- 填入 `ai-workflow/环境真相档案.md` §四。
- 执行 `bash deploy.sh --install-review-addon`，把禁用歧路加入 `code-self-review` §6.9。

**之后** — 回灌习惯自然长：每次踩坑 AI 起草、人审、进档案。

## 5. 怎么证明它有用（量化胜仗）

推广的命门是**第一个能讲的硬故事**。盯这几个指标：

| 指标 | 怎么数 | 目标信号 |
|---|---|---|
| **拦下的假成功次数** | 验证闭环 skill 报告里"未验证完成"的次数 | "本周拦下 N 次旧版本/旧缓存上线" |
| **归因省下的白工** | 归因结论为"非代码层"的次数 | "N 次本来要写代码，归因后发现是配置，省了 X 小时" |
| **取值绕路命中** | code-self-review §6.9 命中数 | 串值/变空类 bug 在 review 阶段就被挡 |
| **档案条目增长** | 回灌候选区合并数 | 档案在长 = 活体没死 |

**第一个周会，讲一个"今天验证闭环拦住一次旧版本上线"的具体故事**——比任何规范宣讲都有用。

## 6. 回应常见质疑

| 质疑 | 回应 |
|---|---|
| "又是一套规范，没人会看" | 它**不是给人背的**。机器能卡的（PR 跑 review、ls-remote 比对）CI 硬卡；要判断的做成 Skill，一键走流程，**遵守比不遵守省事**。 |
| "我们项目不一样" | 不同技术栈可以复用四阶段骨架，但环境真相档案、取值地图、验证命令必须按项目自填。**别硬套具体表格。** |
| "AI 自动写档案会写错" | AI 只起**草稿**，进"回灌候选区"，**人审后才合并**。AI 是抄写员，人是审核关口。 |
| "加四个阶段会变慢" | 慢的是"来回三趟改旧码""为配置问题写一下午代码"。四阶段是把返工提前堵在动手前/上线前。**净省时间。** |
| "取值地图维护成本高" | 它就是把你本来要踩的坑提前记一行。**踩坑即回灌**，不是额外造文档，是把损失变成资产。 |
