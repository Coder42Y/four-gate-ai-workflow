# 四阶段 AI 开发工作流 · 蓝图（规范性）

> **这份是什么**：把某政企合同项目（已脱敏）一个多月的真实 AI coding 实战，提炼成一套**能在同栈项目群推广的强制纪律**。
>
> **和 `AI全栈开发工作流指南.md` 的关系**：那份是**描述性**的——AI 实际做了什么、数千次工具调用怎么分布；这份是**规范性**的——该强制什么、怎么不翻车。两份配套读。
>
> **适用边界（窄普适）**：Spring Boot + Vue3 + 流程引擎（ezgo/工作流）+ UIM 权限 + nacos 配置 + **手工运维 / 无 Flyway / 远端 CI 编译**的政企/中台项目群（同类 Spring/Vue 中台项目）。不是面向全公司任意技术栈——换栈请只取「四阶段」的抽象骨架，具体表格各项目自填。

---

## 0. 一句话信条

> **让"对的做法"成为阻力最小的路径。**
> 机器能判定的硬卡（CI/hook）；要判断的，用 Skill 把"遵守"做得比"不遵守"还省事。绝不靠一份没人看的 PDF 规范。

---

## 1. 为什么是"这一类"项目特别需要纪律

普通项目：代码在本机跑，写对了基本就对了。

这一类项目的三个结构性特征，让"代码看着对"和"线上真的对"之间裂开一条鸿沟：

| 特征 | 后果 | 脱敏实证 |
|---|---|---|
| **决定正确性的信息不在代码里** | nacos 配置、字典码值、组织维度、有没有 Flyway——AI 读穿代码也推不出来 | 扁平查字典把"类别 A"显示成"类别 B"；目标环境无自动迁移，加列即崩 |
| **代码不在本机跑，靠远端 CI 编译 + 手工运维部署** | "改完了"≠"生效了"，旧码/旧 jar/旧缓存可藏在任意一环 | 本地不 push→Jenkins 编的全是旧码；部署 SUCCESS 但 jar 是旧的 |
| **行为由"配置"而非"代码"决定（流程引擎/nacos/UIM）** | 很多"bug"根本不是代码问题，写代码去修反而引入新 bug | 审批按钮占位 key=编辑器没配；引擎不发回调=运维停摆 |

**结论**：这类项目里，AI（和人）最贵的三个错误是——
1. **不知道外部真相就动手**（取错值还浑然不觉）；
2. **为不是代码问题的问题写代码**（白费功夫 + 引新 bug）；
3. **没有线上实证就宣称"修好了"**（假成功上线）。

四个阶段，就是分别堵死这三个错误 + 一道取值纪律。阶段 0 之前可加一个开工前准备动作：把模糊需求追问成任务简报，再进入归因。

## 2. 脊柱：四个阶段（一次任务的生命周期）

```
准备 ─▶ 任务简报 ─┐
                  ▼
       ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
任务 ─▶│ 阶段 0  │─▶│ 阶段 1  │─▶│ 阶段 2  │─▶│ 阶段 3  │─▶ 宣称完成
       │  归因   │   │外部真相 │   │取值地图 │   │验证闭环 │
       └─────────┘   └─────────┘   └─────────┘   └────┬────┘
        是哪一层?      代码外的     钉到唯一正路    无实证不许
        非代码不写码   真相先核对   绕路即违规      说"修好了"
                                                       │
                          回灌：踩坑即写回档案 ◀────────┘（AI 当抄写员）
```

### 前置准备 · 任务简报（开工前先问清楚）

当任务还停留在"帮我做 X / 修一下 Y"时，先触发 `ai-task-preflight`，把目标、现象、期望、边界、上下文、外部依赖、验收证据和风险问清楚。能从代码或文档查到的，AI 自己查；查不到的，一次只问一个问题，并给推荐答案。

产出物是一份「AI 任务简报」。任务简报完成后，才进入阶段 0「归因」。

### 阶段 0 · 归因（动码前先定层）

> **最反直觉、公司级推广价值最高的一道。** 所有人和所有 AI 的本能都是"看到问题直接写代码"——在这类项目里这个本能一半以上是错的。

**动作**：动任何代码前，强制把问题钉到一层，并给证据：

| 层 | 典型真因 | 脱敏实证 |
|---|---|---|
| **代码** | 逻辑/空指针/调用错 | runProcess 裸 axios 无 token → 401（这个才是代码） |
| **配置** | nacos / 流程编辑器 / 平台事件 配置缺失 | 审批按钮占位 key（编辑器没勾命令按钮）；合同不发号（平台事件配置为空）；isSubCompany undefined（网关流程变量没注入） |
| **运维** | 服务存活 / 部署 / 远端同步 | 引擎不自动发回调（事件派发服务停摆）；本地没 push→构建旧码 |
| **数据** | 字典 / 组织 / 权限 配置 | 找不到审核人（岗位没挂到那个部门）；码值对不上 |

**铁律**：**不是"代码"层，就不许写代码**——先去配置/运维/数据层求证。

→ 实现：`skills/attribute-rootcause/SKILL.md`

### 阶段 1 · 外部真相（代码里看不到的真相先核对）

**动作**：动手前核对本项目的外部真相。真相分两种性质，两种机制：

- **结构性真相（稳定，可写死成文档）** → 入库的 `环境真相档案.md`，随仓库走、人人加载。
  - 例：目标环境无自动迁移、appId=<从配置读取>、合同类型是多级字典树、某岗位的组织维度。
- **运行时状态（易变，写死必过期）** → **不写文档，强制现场探针**。
  - 例：这列 ALTER 了没（`information_schema`）？引擎此刻发不发回调？线上 jar 新不新（typeName 探针）？

**铁律**：**结构性真相入库、运行时状态现探；绝不把运行时状态写进文档（必过期），也绝不靠现探去问稳定结构（浪费且易错）。**

→ 实现：`环境真相档案.md`（模板 + 脱敏实例）

### 阶段 2 · 取值地图（不写死 ID 的"真正"机制）

> **挑战一个直觉误区**：很多人以为"不写死 ID"= 别写裸数字 `APP_MAIN_ID`、改成查字典。**这个理解不够，甚至有害。**
>
> 脱敏实证：`getCodeNameByParentCodeType` **走了字典查询**（没写死），却因扁平查多级树 + 数字 value 跨字典复用，把 type=A 类别 A**取成了"类别 B"**；查 foundation 状态字典码值对不上，**静默变空**。**真正的杀手不是"写了字面量"，而是"取值走了一条看似合理、实则错误的路"。** grep 魔法数字的 lint 根本抓不到这两个坑——它们走的是真·API。

**动作**：写任何取外部值的代码前，先查「取值地图」把概念钉到**唯一正路**；表里没有的，**停下来问人，绝不自己发明查法**。

**分工**：
- **「禁裸字面量」** → 已由 `code-self-review` §6.7 硬编码红线覆盖（appId/roleId/menuId 裸数字）。
- **「走对路」** → 本阶段新增，扩进 `code-self-review`（见 `取值地图增补.md`），review 查的是**"有没有绕开正路"**而非"有没有裸数字"。

→ 实现：`取值地图增补.md`（给 code-self-review 加 §6.9），取值地图本体在 `环境真相档案.md` 内。

### 阶段 3 · 验证闭环（反假成功，第一枪打这里）

> **你记忆里密度最高的母题**：以为做完了，线上其实是旧码/旧 jar/旧缓存/引擎没动。

**潜伏链**（旧东西可藏在任意一环）：
```
本地 commit ─(不 push)─▶ 远端 origin ─▶ Jenkins 构建 ─▶ jar/bundle
   ▲                                                        │
   └ 引擎不自动回调 ◀ 浏览器缓存旧 bundle ◀ 部署SUCCESS≠生效 ◀┘
```

**动作**：改动类型 → 必需实证矩阵（详见 Skill）。

**铁律**（嫁接 superpowers `verification-before-completion`）：**"修好了 / 部署成功 / 应该没问题"这类话，没有引用对应实证，一律不许说出口。Evidence before assertions。**

→ 实现：`skills/verify-closure/SKILL.md`

## 3. 双轨记忆 + 分界线

| 去处 | 放什么 | 判据 |
|---|---|---|
| **入仓库档案**（`环境真相档案.md`） | 客观真相 + 红线 | **队友不知道就会弄坏生产 / 取错值** |
| **升团队 Skill** | 诊断方法论 | 可触发、可复用的**做法**（探针、线上验证法） |
| **留个人 memory** | 纯个人效率碎片 | 只是**我个人的手感/捷径**（如某客户Jira图在哪个目录） |

该项目的多条 auto-memory 按此劈分：appId 表、无 Flyway、字典层级、组织维度、Jenkins push-first → **入档案**；typeName 探针、硬编码探针吐全栈、browser_evaluate 线上验证 → **升 Skill**；其余 → 留个人。

## 4. 活体机制（防腐烂）

档案一旦冻结在第一天就开始过期。必须有机制把"刚踩的坑"自动变成"档案里的永久条目"：

1. **踩坑即回灌**：阶段 0 查出非代码真因、或阶段 3 抓到一次假成功，**当场**写回对应资产（取值路→取值地图；结构真相→档案；新验证需求→矩阵）。
2. **AI 当抄写员**：任务收尾，AI 主动产出一条"本次新学到的外部真相"草稿（它刚踩完坑，最便宜的记录者），**人审一下**再合并。出口从"个人机 memory"改成"项目 PR"。
3. **review 卡点**：reviewer 必问一句"这次有没有学到新外部真相？入档案了吗？"——没回灌的 PR 不算完。

> **真相档案不是"写一次的文档"，是"每次踩坑自动长一条的活体"——AI 是默认抄写员，人是审核关口。**

## 5. 推广策略：少数硬卡 + 多数靠 Skill 降阻

| 强制方式 | 卡什么 | 怎么实现 |
|---|---|---|
| **硬卡（CI / hook，机器能判定）** | PR 必跑 `code-self-review`；后端宣称"修好"前 `git ls-remote origin` 比对 HEAD | settings.json hook / CI gate |
| **Skill 降阻（需判断）** | 归因、取值地图、验证矩阵 | 做成触发即引导的 Skill，遵守比不遵守省事 |
| **绝不做** | —— | 不写 50 页 PDF 要求"自觉遵守" |

---

## 6. 落地排序（拿第一个胜仗）

冷启动最忌一上来铺一大套。按 ROI 可证性排：

1. **第一枪 · 验证闭环 Skill** — 痛点密度最高、ROI 最可证。一旦拦下一次"本要当作修好上线的假成功"，就是周会能讲的硬故事（"今天拦住一次旧 jar 上线"）。
2. **第二 · 归因 Skill** — 最反直觉、教育价值最高，但需要前一个建立的信任。
3. **第三 · 取值地图** — 不新建，扩进已有的 `code-self-review`（践行"别重复造"自己）。
4. **底座 · 环境真相档案** — 不是 Skill，是把 auto-memory 去个人化入库，边做边长。
5. **习惯 · 回灌** — 写进项目 CLAUDE.md，每次收尾自动产真相草稿。

**真正新建的 Skill 只有 3 个：开工准备、验证、归因。** 其余是复用 + 文档 + 习惯。

## 7. 资产清单（本文件夹）

```
four-gate-ai-workflow/
├── deploy.sh                      ← ★ 一键部署（Claude/Codex Skills + 项目级模板 + CLAUDE.md 注入）
├── 安装.sh                        ← 旧版（仅全局软链，推荐用 deploy.sh）
├── 四阶段蓝图.md                   ← 本文件（脊柱，规范性总览）
├── 环境真相档案.md                 ← 阶段1底座：模板 + 脱敏实例 + 取值地图本体
├── 取值地图增补.md                 ← 阶段2：给 code-self-review 加 §6.9（示例项目专用）
├── CLAUDE.md片段-回灌与阶段纪律.md     ← 落进项目 CLAUDE.md 的纪律片段（deploy.sh 自动注入）
├── 推广手册.md                     ← 给人看的"为什么/怎么起步"
├── 四阶段工作流.html               ← 可视化展示页
└── skills/
    ├── ai-task-preflight/SKILL.md   ← 前置准备（任务简报）
    ├── verify-closure/SKILL.md      ← 阶段3（第一枪）
    └── attribute-rootcause/SKILL.md ← 阶段0
```

**一键部署**：`bash deploy.sh /path/to/project`（自动：软链 Skills 到 `~/.claude/skills/` 和 `~/.codex/skills/` + 探测技术栈 + 生成环境真相档案 + 生成取值地图指南 + 注入 CLAUDE.md 片段）。`--install-review-addon` 可自动把取值地图增补并进 `code-self-review`，`--check` 查状态、`--uninstall` 卸载。部署后编辑 `ai-workflow/环境真相档案.md` 中的 `[待填]` 项即可。

远程一行安装：

```bash
curl -fsSL https://raw.githubusercontent.com/Coder42Y/four-gate-ai-workflow/master/bootstrap.sh | bash -s -- --with-review-addon /path/to/project
```
