# 四阶段 AI 开发工作流 · 蓝图（自然语言触发版）

> **这份是什么**：一套面向真实业务开发的 AI 协作工作流。阶段是内部控制模型，不是要求开发者背诵的操作口令；日常使用靠自然语言触发，由 Codex skills 和项目 `AGENTS.md` 自动接管。
>
> **适用边界**：适合存在外部配置、权限系统、流程引擎、数据字典、远端构建、手工部署、数据库迁移差异等外部依赖的业务系统。不同技术栈可以复用骨架，具体环境档案、取值地图和验证命令必须按项目自填。

---

## 0. 设计信条

> **让“对的做法”成为阻力最小的路径。**

机器能判定的用 CI/hook 卡；需要判断的，用 skill 和 `AGENTS.md` 做成默认路径。开发者不需要记阶段术语，只要正常描述任务，Codex 根据意图触发对应动作。

---

## 1. 这类项目为什么需要工作流

普通项目里，代码在本机跑，写对了基本就对了。真实业务系统不是这样：

| 结构特征 | 后果 | 常见表现 |
|---|---|---|
| 正确信息不全在代码里 | 配置、字典、组织维度、迁移状态，AI 读穿代码也推不出来 | 调了合法 API 仍取错值；目标环境缺迁移导致运行失败 |
| 代码不只在本机生效 | “改完了”不等于“目标环境已生效” | 本地没 push、CI 编旧码、部署包没换、前端缓存旧产物 |
| 行为由配置或外部系统决定 | 很多“bug”不是代码问题 | 权限没配、流程节点没发布、配置中心未刷新、外部服务停摆 |

工作流要堵住三类高成本错误：

1. **不知道外部真相就动手**。
2. **为不是代码问题的问题写代码**。
3. **没有可复核实证就宣称完成**。

## 2. 自然语言触发模型

`deploy.sh` 会把下面的路由注入项目 `AGENTS.md`。Codex 每次从业务 repo 启动时读取它，因此用户不需要显式说“阶段 0/1/2/3”。

| 自然语言 / 当前动作 | 自动进入 | 必须产出 |
|---|---|---|
| “帮我做/修一下/实现一下”，但目标、边界、复现或验收不清 | 开工准备 | AI 任务简报 |
| “不对/报错/没生效/找不到人/按钮异常/为什么这样” | 归因 | 代码/配置/运维/数据定层 + 证据 |
| 准备读取外部 ID、字典、状态、角色、组织、配置值 | 取值地图 | 唯一正路；没有正路就停下问人 |
| 涉及配置中心、流程引擎、权限、DB 迁移、部署、缓存 | 外部真相 | 结构真相读档案；运行时状态现场探针 |
| 准备说“修好了/完成了/可以合/部署成功” | 验证闭环 | 按改动类型列出可复核实证 |
| 任务里发现新配置坑、新取值路径、新验证探针 | 回灌 | 档案候选条目 |

## 3. 内部脊柱：四个阶段

```
模糊任务 ─▶ 任务简报 ─┐
                      ▼
           ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
任务事实 ─▶│  归因   │─▶│外部真相 │─▶│取值地图 │─▶│验证闭环 │─▶ 可宣称完成
           └─────────┘   └─────────┘   └─────────┘   └────┬────┘
            哪一层?       代码外事实     唯一正路       可复核实证
            非代码不写码   先核对         不自创查法     不用“应该”
                                                           │
                              回灌：踩坑写回档案 ◀────────┘
```

### 开工准备 · 任务简报

当任务还停留在“帮我做 X / 修一下 Y”时，触发 `ai-task-preflight`。它把目标、现象、期望、边界、上下文、外部依赖、验收证据和风险收敛成任务简报。能从代码或文档查到的，AI 自己查；查不到的，一次只问一个关键问题，并给推荐答案。

### 阶段 0 · 归因

动任何代码前，先把问题钉到一层，并给证据：

| 层 | 典型真因 | 常见表现 |
|---|---|---|
| 代码 | 逻辑、空指针、调用错 | 接口必 401、异常栈指向明确代码路径 |
| 配置 | 配置中心、流程编辑器、平台事件、权限配置缺失 | 按钮占位符、流程变量缺失、事件未配置 |
| 运维 | 服务存活、部署、远端同步 | 回调服务停摆、目标环境未更新、构建使用旧代码 |
| 数据 | 字典、组织、角色、权限数据 | 找不到处理人、码值对不上、列不存在 |

**铁律**：不是代码层，就不写代码，先去对应层求证。

实现：`skills/attribute-rootcause/SKILL.md`

### 阶段 1 · 外部真相

动手前核对代码外的项目事实。真相分两类：

- **结构性真相**：稳定，可入库，如迁移机制、配置来源、字典层级、组织维度、权限系统入口。
- **运行时状态**：易变，不写死，如目标环境当前版本、某列此刻是否存在、某服务此刻是否在跑。

**铁律**：结构性真相写入 `环境真相档案.md`；运行时状态现场探针。

实现：`环境真相档案.md`

### 阶段 2 · 取值地图

“不写死 ID”不是只禁止裸数字。更常见的坑是：代码调用了真实 API，但因为取值路径不对，最终取错值。

写任何取外部值的代码前，先查取值地图，把概念钉到唯一正路。表里没有的概念，停下问人，不自创查法。

实现：取值地图本体在 `环境真相档案.md` §四；review 增补在 `取值地图增补.md`。

### 阶段 3 · 验证闭环

“改完”到“目标环境生效”之间可能藏旧码、旧包、旧缓存或外部流程未触发：

```
本地 commit ─(不 push)─▶ 远端 origin ─▶ CI 构建 ─▶ 部署包/前端产物
   ▲                                                        │
   └ 外部流程未触发 ◀ 浏览器缓存旧产物 ◀ 部署 SUCCESS ≠ 生效 ◀┘
```

**铁律**：“修好了 / 部署成功 / 应该没问题 / 已完成”这类结论，必须先引用对应实证。

实现：`skills/verify-closure/SKILL.md`

## 4. 资产分工

| 去处 | 放什么 | 判据 |
|---|---|---|
| `AGENTS.md` / `CLAUDE.md` | 项目级自然语言路由和硬规则 | 每次会话都要加载 |
| `环境真相档案.md` | 客观真相、取值地图、回灌候选 | 队友不知道就会弄坏目标环境或取错值 |
| Skills | 可触发、可复用的做法 | 开工澄清、归因、验证 |
| `code-self-review` 增补 | 可 grep 的取值歧路 | 把“走错路”挡在 review 阶段 |

---

## 5. 活体机制

档案冻结在第一天就会过期。每次任务收尾，如果发现新的外部真相、新取值正路、新部署探针或新验证要求，AI 起草回灌候选，人审后合并。

```markdown
## [回灌候选] <一句话标题>
- 归类：部署链路 / DB / 取值地图 / 流程引擎 / 配置中心 / 前端 / 协作
- 现象：
- 真因：
- 正解：
- 证据：
- 状态：待人审
```

---

## 6. 落地排序

1. **第一枪：验证闭环**。最快证明价值，能直接拦住“旧版本/旧缓存/未部署却宣称完成”。
2. **第二：归因**。把“看到问题就写代码”的反射改成“先定层”。
3. **第三：取值地图**。把“走真 API 但取错路”的隐性 bug 放进 review。
4. **底座：环境真相档案**。把项目事实从个人经验变成仓库资产。
5. **习惯：回灌**。每次踩坑让档案多一条可复用事实。

真正新建的日常 Skill 只有 3 个：开工准备、验证、归因。安装入口只负责把工作流接入当前业务 repo，不参与日常编码判断。
