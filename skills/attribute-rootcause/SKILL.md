# 归因 Skill · 动码前先定层

> **SSOT**：与 `ai-workflow/四阶段蓝图.md` 的归因规则一致。这是工作流里最靠前、也最反直觉的一道。

## 一、为什么要有它

所有人和所有 AI 的本能都是：**看到问题 → 直接写代码修**。

但在外部依赖较多的业务系统里（配置中心 / 权限系统 / 流程引擎 / 数据字典 / 组织角色），很多"bug"根本不是代码问题。为它写代码 = 白费功夫 + 可能引入真的新 bug + 真因还在。

同样是"功能不对"，真因可能散在四层：

| 现象 | 真因层 | 写代码能修吗 |
|---|---|---|
| 接口必 401 | **代码**（请求未带 token 或鉴权逻辑错误） | ✅ 这个才该写代码 |
| 流程变量 undefined | **配置**（流程变量没注入） | ❌ 先补配置/发布 |
| 按钮是占位 key | **配置**（按钮/i18n/权限配置缺失） | ❌ 先补配置 |
| 回调没有触发 | **运维**（事件服务停摆或外部系统未运行） | ❌ 先查服务和链路 |
| 状态没有自动流转 | **配置**（平台事件或规则为空） | ❌ 先补事件/规则 |
| 找不到处理人 | **数据**（岗位/角色/组织维度未配置） | △ 取法可能要改，但先确认数据维度 |

## 二、铁律

> **没把问题钉到具体一层、且给出证据之前，不许写一行修复代码。**
> **定层结论不是"代码"时，不许写代码——去对应层求证/动手。**

## 三、四层判定法

对现象逐层问，带证据排除，直到锁定一层：

### 1. 代码层（最后才默认它）
- 问：同样的输入，代码路径会必然产生这个错吗？能在逻辑里复现吗？
- 证据：读到具体出错的代码行 / 栈帧 / 空指针来源。
- ⚠️ **不要因为"我会写代码"就默认是代码问题**——这是最贵的偏见。

### 2. 配置层（配置中心 / 流程引擎 / 权限系统 / 平台事件）
- 问：这个行为是不是由某处**配置**决定的？那处配了吗、配对了吗？
- 证据：
  - 流程：进编辑器看节点是否勾了命令按钮、网关变量是否注入、流程"事件"是否为空。
  - 配置中心：看对应 key / namespace / group 的值（只读读法见档案）。
  - 权限系统：看按钮/菜单/角色权限是否写入、appId 是否对（appId=<从项目配置读取>）。
- 高频：**占位 key、流程变量 undefined、事件未触发——优先查配置层。**

### 3. 运维层（服务存活 / 部署 / 远端同步）
- 问：相关服务此刻活着吗？代码真同步到执行处了吗？
- 证据：
  - `git ls-remote origin` vs 本地 HEAD（不一致=没 push，构建旧码）。
  - 事件派发/引擎服务是否在跑（停摆=回调不发，非代码）。
  - 后端包/前端产物是否真换新（版本接口、构建号、业务探针）。
- 高频：**回调不发(服务停摆)、改动没生效(没 push)——运维层。**

### 4. 数据层（字典 / 组织 / 权限）
- 问：取值依赖的字典/组织/权限数据，配齐了吗、维度对吗？
- 证据：
  - 实查字典码值是否存在、层级对不对。
  - 组织维度：该岗位挂在公司级还是部门级、目标部门有没有该岗。
  - `information_schema` 看列在不在。
- 高频：**找不到处理人(岗位/角色维度)、码值对不上(字典)、列不存在(迁移未执行)——数据层。**

## 四、执行流程

1. **复述现象**（一句话，含可观测细节）。
2. **逐层排除**：按 §三 1→2→3→4，每层给"是/否 + 证据"。拿不准就实际去查（读代码、看配置、ls-remote、查 DB），**不靠猜**。
3. **锁定一层 + 证据**，输出 §五 结论。
4. **非代码层** → 给出该层的动作（配什么、找谁、补什么数据），**不写修复代码**。
5. **代码层** → 才核对外部真相和取值地图，然后写代码，并在收尾时验证。

## 五、归因结论模板

````markdown
# 归因结论

**现象：** <一句话 + 可观测细节>

| 层 | 是否真因 | 证据 |
|---|---|---|
| 代码 | 否 | 代码路径无此分支，已读 XxxService:42 |
| 配置 | **是** | 流程编辑器 UserTask_2 未勾命令按钮，i18N 缺失 |
| 运维 | 否 | 服务在跑，ls-remote==HEAD |
| 数据 | 否 | 字典/组织已核对 |

**根因层：配置**
**下一步：** 去流程编辑器逐节点勾命令按钮并重新发布；**本问题不写代码**。
**（若代码层）** → 核对外部真相 → 查取值地图 → 写码。
````

## 六、给 agent 的元规则

- **默认偏见相反**：先假设"不是代码问题"，用证据逼自己排除配置/运维/数据，最后才落到代码。
- **一个现象可能多层叠加**（如发号：平台事件空[配置] + 业务系统缺少回调端点[代码]）——都列出，标主次。
- **拿不准就去查**，别在结论里写"可能是"。
- 归因完若是代码层，**交棒给外部真相/取值地图**；若非代码层，**产出动作清单交人/运维**，不越界写码。
