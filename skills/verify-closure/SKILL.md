# 验证闭环 Skill · 反假成功（阶段 3）

> **SSOT**：本 skill 与 `ai-workflow/四阶段蓝图.md` 阶段 3 一致。它把"完成"从"我改完了"重定义为"我有线上实证"。

## 一、铁律（违反即视为本 skill 失败）

> **没有引用下表对应实证之前，"修好了 / 部署成功 / 应该没问题 / 已完成"这类话一律不许说出口。Evidence before assertions.**

拿不准时默认"未验证"，把缺的实证列出来请人补，**绝不用"应该""理论上""一般来说"补位**。

## 二、为什么需要它（假成功潜伏链）

这类项目里"改完"和"生效"之间裂着一条链，旧东西可藏在任意一环：

```
本地 commit ─(不 push)─▶ 远端 origin ─▶ Jenkins 构建 ─▶ jar/bundle
   ▲                                                        │
   └ 引擎不自动回调 ◀ 浏览器缓存旧 bundle ◀ 部署SUCCESS≠生效 ◀┘
```

**该验证哪几环，取决于你改了什么。** 这就是下面的矩阵。

## 三、执行流程

### Step 1 — 判定改动类型

看本次 diff 涉及什么，可能命中多类（如"前端 + 后端 + DB"三类全跑）：

```bash
git diff --name-only HEAD
```

| 命中 | 改动类型 | 走第几行矩阵 |
|---|---|---|
| `*.vue` `*.ts` `*.scss` `frontend-app/**` | 前端 | §四 A |
| `*.java` `*.xml`(mapper) `backend-module/**` | 后端 | §四 B |
| `*.sql` / 引用了新列 | DB 结构 | §四 C |
| `*.bpmn` / 流程相关 / 审批节点 / 回调 | 流程引擎 | §四 D |
| nacos 配置项（dataId/group）变更 | nacos 配置 | §四 E |
| UIM 菜单/按钮/角色/岗位权限变更 | UIM 权限 | §四 F |

### Step 2 — 逐类取实证

对每个命中类型，跑 §四 对应行的"必需实证"。**每条实证必须有可粘贴的观测结果（命令输出 / 接口返回 / 截图），不是"我看了一眼"。**

### Step 3 — 出验证报告 + 回灌

按 §五 模板输出。报告里任何一格是"未取得"，结论即"未验证完成，不可宣称修复"。最后走 §六 回灌。

## 四、改动类型 → 必需实证矩阵

### A. 前端改动

| 必需实证 | 怎么取（脱敏实例） | 不做的后果 |
|---|---|---|
| 1. 远端已收到新码 | `git ls-remote origin <branch>` == 本地 `git rev-parse HEAD` | 不 push→CI 编旧码 |
| 2. 构建产物已更新 | 触发 CI / 看 bundle hash 变了 | 部署的是旧 bundle |
| 3. **清缓存重登后**线上验证 | 硬刷（Ctrl+Shift+R）/ 无痕窗口重新登录 → `browser_evaluate` 调接口 / 截图比对 | 浏览器缓存旧 bundle，"看不到改动"（脱敏案例：工单 A/B） |

> 实战教训 `example-case`：index.html 无缓存头，**复验前必先清缓存 + 重登**，否则你看的是旧 bundle。

### B. 后端改动

| 必需实证 | 怎么取（脱敏实例） | 不做的后果 |
|---|---|---|
| 1. **远端已收到新码** | `git ls-remote origin <branch>` 的 HEAD == 本地 HEAD | 本地 commit 没 push，Jenkins 从内部 Git 远端拉的是旧码，部署 SUCCESS 但全是旧逻辑（`example-case`） |
| 2. 构建成功且是新构建号 | 轮询 Jenkins 构建号 / `result=SUCCESS` / 读 `consoleText` | 构建失败却以为成功 |
| 3. **jar 真的换新了** | 30 秒 typeName 探针：开"类别 A 合同(typeA)"详情，看 `typeName`="类别 A"(新) vs "类别 B"(旧)（`example-case`） | 前端发了但后端 jar 是旧的，"全没改好" |
| 4. 接口实测 | `browser_evaluate` 带 `Authorization`+`x-tenant-id` 调 `/backend-service/...`，看返回 | —— |

> 别用 `pkill -f backend-service.jar` 重启（会自杀 exec shell）。token ~5min 过期会致详情整片变空(401)，别误判成"改坏了"。

### C. DB 结构改动（加列等）

| 必需实证 | 怎么取（脱敏实例） | 不做的后果 |
|---|---|---|
| 1. **列真的建了** | `information_schema.columns` 实查该列存在 | 目标环境 **无 Flyway**，迁移脚本不会自动跑，"加列+代码引用该列"部署即崩 BUSINESS_ERROR_CODE（mapper 查不存在的列，`example-case`） |
| 2. ALTER 已手工执行 | 在目标环境手工跑 `ALTER TABLE ... ADD COLUMN`（**普通 ADD，不能用 `IF NOT EXISTS`**——需按目标环境 MySQL 方言实测，`example-case`） | 列没建，代码引用即异常 |
| 3. 顺序铁律 | **先 ALTER 补列，再上引用该列的代码**——顺序反了必崩 | —— |

### D. 流程引擎改动（审批/回调/节点）

| 必需实证 | 怎么取（脱敏实例） | 不做的后果 |
|---|---|---|
| 1. **先确认是不是代码问题**（回阶段 0） | 很多流程"bug"是配置/运维：按钮占位 key=编辑器没勾命令按钮（`example-case`）；引擎不发回调=事件派发服务停摆（`example-case`） | 为非代码问题写代码 |
| 2. 节点 i18n/命令按钮已在**编辑器逐节点配并发布** | 进流程编辑器核对，重新发布 | 按钮显示占位 key |
| 3. 流程变量已注入 | 走到该网关/节点实测，不报 `No such property`（`example-case`） | 网关变量缺失流程卡死 |
| 4. **手动 POST 回调→状态翻转**实证链路 | 手动 POST 回调端点，看业务状态变 APPROVED；引擎自动回调要单独验（可能运维停摆） | 以为链路通、其实引擎没发 |
| 5. 审批人取得到 | 走到审批节点看有没有人；岗位维度用带回退的取法（`selectApproversWithFallback`，`example-case`） | "找不到审核人"流程卡死 |

### E. nacos 配置改动

| 必需实证 | 怎么取 |
|---|---|
| 1. 配置已发布到对的 dataId/group/namespace | nacos 控制台核对（只读读法见档案） |
| 2. 应用已刷新该配置 | 看应用日志有无 refresh / 实测受该配置影响的行为 |
| 3. 区分"配置层"问题 | 若行为没变，先确认是配置没生效还是代码没读对，别直接改代码 |

### F. UIM 权限改动（菜单/按钮/角色/岗位）

| 必需实证 | 怎么取（脱敏实例） |
|---|---|
| 1. funcPerm 写入成功 | UIM REST 写入坑：add 必带 appId、code 全局唯一、id 在 msg 里、字符串、token 5min 过期（`example-case`） |
| 2. 前端按既有机制读权限 | 用既有 `$hasBtnPermission`（读 route.meta.buttons），别新建 v-hasPerm；菜单走动态路由别写死 |
| 3. 用目标角色账号实登验证 | 换该角色登录，看菜单/按钮真出现 |

## 五、验证报告模板（强制）

````markdown
# 验证闭环报告

**改动范围：** <一句话，如"fix 分支：合同变更后编号，含后端+DB">
**命中类型：** 后端 ✅ / DB ✅ / 前端 ❌ / 流程 ❌ / nacos ❌ / UIM ❌

## 实证清单

| # | 类型 | 必需实证 | 观测结果 | 状态 |
|---|---|---|---|---|
| 1 | 后端 | ls-remote==本地HEAD | `abc123` == `abc123` | ✅ |
| 2 | 后端 | jar 换新(typeName探针) | typeName="类别 A"(新) | ✅ |
| 3 | DB | 列存在(information_schema) | new_contract_no 存在 | ✅ |
| 4 | DB | 已手工ALTER | 已在目标环境执行 | ✅ |

**结论：** <已验证完成，可宣称修复 / 未验证完成（缺 #N），不可宣称修复>

## 仍缺的实证（如有）
- <列出，请人补或说明为何无法取>
````

## 六、回灌（收尾必做）

验证过程中若发现一条**新的外部真相**（新的旧码藏身处、新的探针、新的配置坑），按蓝图 §4 当场起草一条，追加到 `环境真相档案.md` 候选区，交人审：

```markdown
## [回灌候选] <一句话标题>
- 类型：结构真相 / 取值路 / 验证矩阵新行
- 现象 / 真因 / 正解：
- 来源：本次任务 / commit / 工单号
```

## 七、给 agent 的元规则

- **不许用"应该/理论上"补位实证。** 没取到就写"未取得"。
- **改了多类就跑多类**，别只验最显眼的一类。
- **token 过期致页面变空(401) ≠ 改坏了**——先重新登录再判。
- **部署 SUCCESS ≠ 生效**——构建成功只是必要条件，jar/bundle/缓存/引擎都要单独验。
- **别被"提交人/流程节点"骗**——只认线上实测的内容（如 typeName 探针）。
