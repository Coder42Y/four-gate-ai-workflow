# AI 任务预备 Skill · 开工前先问清楚

> **定位**：这是四阶段工作流的**前置准备动作**，发生在阶段 0「归因」之前。它解决的是：同事还没准备好需求、边界、验收和上下文，就直接让 AI 开始写代码。

## 一、目标

把一句模糊任务，追问成一份能交给 AI 执行的「任务简报」。

AI 不应急着写方案或代码，而是先确认：

| 维度 | 要问清什么 |
|---|---|
| 目标 | 要解决什么业务问题？成功后谁能看到什么变化？ |
| 现象 | 当前具体哪里不对？有没有可复现路径、截图、接口返回、日志？ |
| 期望 | 正确行为是什么？谁定义它？有没有反例？ |
| 边界 | 本次做什么、不做什么？哪些模块/仓库只读？ |
| 上下文 | 相关页面、接口、表、配置、流程节点、任务/缺陷链接在哪里？ |
| 外部依赖 | 是否涉及配置中心、流程引擎、权限、字典、组织、DB 迁移、部署环境？ |
| 验收 | 什么证据能证明完成？本地、测试、线上分别怎么验？ |
| 风险 | 哪些动作危险（写库、部署、删数据、权限变更）？需要谁确认？ |
| 未知项 | 哪些信息缺失？哪些可由 agent 自己查，哪些必须问人？ |

## 二、执行铁律

> **没有形成任务简报前，不进入实现。**
> **能通过读代码/文档/配置查到的，不问人；查不到的，一次只问一个问题。**
> **每个问题都要给推荐答案或推荐选择，降低用户回答成本。**

## 三、提问方式

1. 先复述用户的原始任务，用一句话说清你理解的目标。
2. 快速扫描已有上下文：README、项目文档、相关文件、错误信息、任务/缺陷内容、测试。
3. 把能查到的信息先填进任务简报草稿。
4. 对缺口按依赖顺序追问，**一次只问一个问题**。
5. 每个问题必须带「推荐答案」，格式：

```markdown
我还缺一个关键判断：<问题>

推荐答案：<基于当前上下文的建议>
为什么问：<这个答案会影响范围/实现/验证的哪一部分>
```

6. 当关键分支都收敛后，输出 §五 的任务简报，并交棒到阶段 0。

## 四、追问顺序

按下面顺序走，不要随机散问：

1. **目标与期望**：成功后具体变化是什么？
2. **复现与证据**：当前现象怎么稳定观察？
3. **范围边界**：本次允许动哪些模块，不动哪些？
4. **外部依赖**：是否涉及配置/权限/流程/字典/DB/部署？
5. **验收标准**：什么证据算完成？
6. **风险确认**：是否有写库、部署、删数据、权限变更等危险动作？
7. **未知项处理**：哪些由 AI 继续查，哪些需要人补？

## 五、任务简报模板

```markdown
# AI 任务简报

**目标：** <一句话说明要达成的业务/工程结果>

**当前现象 / 背景：**
- <可观察现象、截图/日志/接口/任务链接位置>

**期望行为：**
- <正确行为与判定来源>

**范围：**
- 本次包含：<模块/页面/接口/脚本>
- 本次不包含：<明确排除项>

**上下文入口：**
- 代码/文档：<路径>
- 外部系统/配置：<如有>

**外部依赖初判：**
- 配置 / 运维 / 数据 / 权限 / 流程 / DB：<涉及与否，证据>

**验收证据：**
- 本地：<命令/测试/截图>
- 线上或目标环境：<接口/探针/截图/日志>

**风险与确认：**
- <危险动作、需确认事项>

**仍未知：**
- <缺口；注明由 AI 查还是问人>

**下一步：** 进入阶段 0「归因」；如果根因不是代码层，不写代码。
```

## 六、给 agent 的元规则

- 不要用连续十几个问题淹没用户；一次一个，且问题必须影响后续决策。
- 不要问代码里能查到的问题；先读项目文档和相关代码。
- 如果用户只给一句“帮我做 X”，优先问验收标准或复现路径。
- 如果任务明显是新功能，优先问范围边界和目标用户工作流。
- 如果任务明显是 bug，优先问复现路径、当前证据和期望行为。
- 任务简报完成后，不直接实现；先进入阶段 0 做代码/配置/运维/数据归因。
