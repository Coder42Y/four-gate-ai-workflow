四阶段 AI 开发工作流 / 使用说明 先接入业务 repo，再日常调用

HOW TO USE

给同事照着操作的四阶段使用说明

这个页面只解决一个问题：安装以后怎么在真实业务 repo 里调用。现有的 四阶段工作流.html 用来讲方法论；本页用来落地、培训、排错。

最短路径

每个新业务 repo 直接执行 bootstrap，之后从该 repo 启动 Codex。

1
进入业务 repo在项目根目录执行远程 bootstrap
2
生成项目上下文写入 ai-workflow/ 和 AGENTS.md
3
启动 Codex方案发散、开工准备、归因、验证闭环都在业务 repo 里完成
安装入口 接入新 repo 日常调用 生成文件 管理命令 排错 介绍话术
01

每个新业务 repo：执行 bootstrap

给同事复制这一条

在业务项目根目录执行。它会拉取工作流仓库，安装 Codex skills，并把自然语言路由写入当前项目。

terminal
cd /path/to/business-repo
curl -fsSL https://raw.githubusercontent.com/Coder42Y/four-gate-ai-workflow/master/bootstrap.sh -o /tmp/four-stage-bootstrap.sh
bash /tmp/four-stage-bootstrap.sh

它装了什么

  • ~/.four-stage-ai-workflow：缓存本工作流仓库。
  • ~/.agents/skills：Codex 可发现的四个 workflow skills。
  • AGENTS.md：Codex 启动时读取的项目级自然语言路由。
  • 安装完后，业务开发仍然在业务 repo 里做，不需要切到本仓库。
02

每个新业务 repo：接入一次工作流

启动 Codex

注意是业务项目根目录，不是四阶段工作流仓库。

terminal
cd /path/to/business-repo
codex

Claude Code 兼容入口

如果团队里有人继续用 Claude Code，可以单独安装 slash command。

terminal
curl -fsSL https://raw.githubusercontent.com/Coder42Y/four-gate-ai-workflow/master/install-claude-command.sh -o /tmp/four-stage-install-claude-command.sh
bash /tmp/four-stage-install-claude-command.sh
更新缓存拉取或更新 ~/.four-stage-ai-workflow
安装 skills软链到 ~/.agents/skills 与 ~/.claude/skills
注入项目生成 ai-workflow/，注入 AGENTS.md
开始使用重开 Codex 会话后按自然语言触发
03

日常 coding：用自然语言触发

场景 直接对 Codex 说 作用
还在想方案 帮我想想这个功能怎么设计，先别写代码 触发 ai-task-brainstorm，先比较候选方案并给推荐路径。
需求还模糊 开始前准备：帮我把这个需求问完整 触发 ai-task-preflight，先形成 AI 任务简报。
看到一个 bug 先归因，不要急着写代码。现象是：... 触发动码前归因，先判定代码 / 配置 / 运维 / 数据是哪层。
要完整处理一个任务 按工作流处理这个任务：... 先发散/收敛方案，再形成任务简报、归因、核对外部真相、查取值地图，最后验证闭环。
准备收尾 验证闭环：这次改动是否真的生效？ 触发 verify-closure，没有实证不能说修好了。
提交前自查 提交前做 code-self-review，并检查取值地图 结合已有 code-self-review 检查取值绕路。
使用边界：同一个业务 repo 接入后，日常 coding 不需要重复安装。换到另一个新业务 repo，第一次在项目根执行一次 bootstrap。
04

安装后会看到哪些文件

AGENTS.md

Codex 启动时读取的项目级路由：按用户自然语言自动触发方案发散、澄清、归因、取值地图、验证和回灌。

ai-workflow/环境真相档案.md

项目级事实库。安装后必须搜索 [待填]，把部署链路、数据库真相、外部 ID、取值地图补齐。

ai-workflow/取值地图指南.md

指导团队把外部值钉到唯一正路，防止“没有硬编码但走错 API”的隐性 bug。

ai-workflow/四阶段蓝图.md

规范性总览，适合想理解完整设计的人读。

ai-workflow/四阶段工作流.html

方法论介绍页，适合团队宣讲。

ai-workflow/四阶段使用说明.html

本页。供同事第一次使用时照着做。

05

管理命令

查看状态

检查全局 skills 和当前项目是否已接入。

terminal
~/.four-stage-ai-workflow/deploy.sh --check .

重新执行 bootstrap

工作流更新或项目文件损坏时，在业务 repo 根目录重新运行。

terminal
curl -fsSL https://raw.githubusercontent.com/Coder42Y/four-gate-ai-workflow/master/bootstrap.sh -o /tmp/four-stage-bootstrap.sh
bash /tmp/four-stage-bootstrap.sh

安装 review 增补

如果已有 code-self-review，可单独补取值地图 §6.9。

terminal
~/.four-stage-ai-workflow/deploy.sh --install-review-addon

卸载当前项目注入

移除全局 skills，并可选择删除当前项目的 ai-workflow/。

terminal
~/.four-stage-ai-workflow/deploy.sh --uninstall .
06

常见问题

现象 原因 处理
Codex 没读到项目路由 不是从业务 repo 根目录启动，或 AGENTS.md 未注入。 回到业务 repo 根目录，执行 ~/.four-stage-ai-workflow/deploy.sh --check .，然后重启 Codex。
误装到 workflow repo 执行 bootstrap 时所在目录不是业务 repo。 退出后 cd 到业务 repo 根目录，再执行 bootstrap。
code-self-review 增补未安装 本机没有安装 code-self-review，或路径不在默认候选里。 主工作流已完成；需要时用 CODE_SELF_REVIEW_SKILL=/path/to/SKILL.md ~/.four-stage-ai-workflow/deploy.sh --install-review-addon。
skills 没立即触发 Codex / Claude Code 可能还没重新扫描 skills。 重开会话，再用“帮我想想”“开始前准备”“先归因”“验证闭环”等触发词。
同一个 repo 是否要重复装 不需要。 同一业务 repo 日常直接调用；只有工作流更新或文件损坏时才重跑。
07

给同事介绍时可以这样说

一句话

这不是让你切到另一个 repo 开发，而是把一套 AI 编程纪律注入到你的业务 repo：先问清楚、先归因、查外部真相、走唯一取值路径、拿证据再说完成。

三句话

  1. 每个新业务 repo 执行一次 bootstrap。
  2. 从业务 repo 根目录启动 codex。
  3. 之后所有 coding 都在业务 repo 里，用自然语言触发四阶段。
