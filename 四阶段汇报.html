面向真实业务开发中常见的外部依赖、环境差异、配置行为和验证缺口，把 AI 协作拆成可执行的工作流：方案发散、任务准备、归因、外部真相、取值地图、实现、验证闭环、回灌档案。

01

要解决的不是“AI 不会写代码”

在这类项目里，主要风险来自代码外事实不可见、跨层取值不一致、环境差异未确认，以及缺少可复核的完成证据。

失败模式

错误归因

配置、数据、权限、流程定义、目标环境差异被误判成代码 bug，导致 AI 在错误层面持续修改。

失败模式

取值断链

页面展示名、前端 value、接口参数、后端枚举、数据库字段不是同一个值，调用真实 API 也可能走错路径。

失败模式

口头闭环

只有“已修改”的说明，没有测试命令、接口返回、页面路径、日志证据和未覆盖风险，后续排查无法复用。

02

从业务仓库接入到项目内约束的注入链路

04

阶段契约：每一步都有输入、动作和产物

四阶段的重点不是流程图本身，而是每个阶段都规定了 AI 可以做什么、必须交付什么、什么时候不能继续。

阶段
输入
AI 动作
产物 / 停止条件
前置：方案发散
粗想法、改造方向、想做成可复用能力的诉求。
比较 2-4 个方案，按复用性、接入成本、维护成本、验证难度做取舍。
输出推荐路径；没有推荐和取舍理由，不进入实现。
开工准备：任务简报
一句话需求、缺陷描述、截图、日志或不完整验收口径。
能查的信息先查，查不到才一次问一个关键问题，并给推荐答案。
输出目标、边界、上下文、外部依赖、验收证据和风险。
阶段 0：归因
现象、报错、复现路径、最近变更。
按页面、接口、服务、SQL、数据、配置、权限、环境逐层排查。
输出归因结论；若是配置/数据/环境问题，停止写码。
阶段 1：外部真相
归因怀疑点和项目事实档案。
核对配置中心、数据库、流程定义、权限系统、CI/CD、目标环境。
输出外部事实清单；事实缺失时不能靠猜继续。
阶段 2：取值地图
业务字段、枚举、接口参数、查询条件。
追踪 UI 展示名、前端 value、API payload、DTO、枚举、SQL、DB 字段。
输出唯一正路；存在多套取值时先统一映射。
阶段 3：验证闭环
代码 diff、配置变更、验证目标。
执行测试、构建、接口请求、页面路径、日志核对。
输出证据和残余风险；无证据不能宣称完成。

调试链路：订单状态筛选不生效

下面不是示意口号，而是这套工作流要求 AI 在真实 bug 里输出的链路级排查结果。

现象：选择“已完成”后列表仍返回全部订单

ROOTCAUSE TRACE
链路层级
实际观察
判断
下一步
页面组件
下拉框 label 为“已完成”，value 为 completed。
页面交互正常，参数确实被选择。
继续查请求参数。
接口请求
GET /api/orders?status=completed
前端传参存在，不是按钮事件失效。
继续查后端 DTO 和枚举。
后端枚举
DTO 期望 OrderStatus.FINISHED。
跨层取值不一致，转换失败后 status 为空。
建立映射或统一 API contract。
SQL 条件
<if test="status != null"> 才拼接筛选条件。
转换失败导致查询条件被跳过。
补测试覆盖空值和有效值。
数据库字段
实际存储值为 DONE。
页面值、后端枚举、DB 值三套命名。
写入取值地图，避免再次猜测。

请沿页面选择值、接口参数、DTO、枚举、

哪一层有证据，哪一层只是推测。
# 触发取值地图

06

验证闭环：把“完成”改成证据集合

阶段 3 要求把结果写成可复核的证据，而不是停在“我改完了”。证据越具体，后续交接和回滚成本越低。

1
命令证据单测、lint、构建、相关模块测试的命令和结果。
2
接口证据请求 URL、参数、响应摘要、关键字段断言。
3
页面证据用户路径、筛选条件、页面结果和异常分支。
4
日志证据服务日志、SQL 日志、错误栈、CI/CD 或目标环境输出。
5
未覆盖范围说明哪些环境、角色、数据量或流程分支尚未验证。
6
回灌记录把新增事实写回环境档案或取值地图。
# 验证闭环提示词
验证闭环：这次改动是否真的生效？
请按“命令证据 / 接口证据 / 页面路径 / 日志证据 / 未覆盖风险 / 回灌内容”输出。
没有执行过的验证不要写成已通过。
07

推广边界和验收口径

这套工作流适合先在一个真实业务项目里试点。不同技术栈可以复用阶段思想，但环境档案、取值地图和验证命令必须按项目重填。

适用

外部依赖重的业务系统

配置中心、权限系统、流程引擎、数据字典、CI/CD、目标环境差异等特征越明显，收益越直接。

试点

先跑真实任务

拿真实需求和 bug 试点，观察是否减少错误归因、取值误判和无证据交付，而不是只看安装是否成功。

验收

看阶段产物是否留下来

每次任务结束后应能看到归因、外部事实、取值地图、验证证据和回灌记录，而不是只有代码 diff。

汇报结论

四阶段的工程价值在于把 AI 辅助开发约束成可复核流程：先确认任务和事实，再决定是否写代码；写完以后用证据关闭，并把新增事实沉淀回项目。
